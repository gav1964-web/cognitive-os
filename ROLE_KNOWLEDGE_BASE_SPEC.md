# Role Knowledge Base Spec

## Purpose

Knowledge Base is not a flat encyclopedia. It is a set of role-facing APIs used by Cognitive OS roles to make bounded engineering decisions.

Each KB record may be consumed by one or more roles:

- `project_analyzer`: stack/archetype signals, entrypoints, source-scope policy, evidence scoring.
- `architect`: project archetypes, target architecture shapes, first-slice recipes, risks, mitigations.
- `spec_writer`: input/output contracts, acceptance criteria, clarification patterns.
- `implementer`: implementation recipes, adapters, safe skeletons, patch constraints.
- `tester`: contract tests, negative tests, fake transports, fixture policies.
- `reviewer`: blocking risks, security/reproducibility/idempotency checks.
- `researcher`: knowledge gaps, external source digests, candidate KB records.

Legacy records without explicit `role_scope` are assigned deterministic defaults by `runtime/role_knowledge.py`.

## Record Semantics

KB records are advisory interfaces, not ground truth and not executable code.

Required invariants:

- facts must be backed by evidence;
- judgments are evaluated separately from facts;
- `teacher_reference` is a checked orientation point, not ground truth;
- records do not mutate runtime state;
- records do not execute arbitrary code;
- role outputs cite matched record ids and matched evidence.

## Evidence Strength

Records may carry `evidence_strength`:

- `weak`: candidate or early observation;
- `medium`: lesson or reviewed heuristic;
- `strong`: active rule/pattern with known evidence;
- `verified`: repeatedly confirmed and accepted.

If absent, runtime assigns a conservative default.

## Admission Policy

New KB knowledge enters through staged candidates:

```text
observation
-> KB candidate
-> several confirmed cases
-> external teacher/corrector approval
-> Codex/developer approval
-> human merge into active KB
```

Automatic self-promotion is forbidden. A candidate may be generated by code, but active KB merge requires review.

Minimum default gate:

- at least 3 confirmed cases;
- `teacher_approved=true`;
- `codex_approved=true`;
- facts separated from judgments;
- source evidence retained.

Implemented contract:

- `runtime/role_knowledge.py`
- `runtime/knowledge_admission.py`
- `tools/knowledge_candidates.py`

## Candidate Storage

Staged candidates are persisted under:

```text
artifacts/knowledge_candidates/<candidate_id>.json
```

The storage is an audit/review queue, not an active KB location. Runtime does not load these files as active rules.

Review summary:

```bash
python tools/knowledge_candidates.py --root . report
python tools/knowledge_candidates.py --root . report --grouped
python tools/knowledge_candidates.py --root . review
```

Manual write:

```bash
python tools/knowledge_candidates.py --root . write --candidate candidate.json
```

Review actions:

```bash
python tools/knowledge_candidates.py --root . approve-teacher --candidate-id kbc_xxx
python tools/knowledge_candidates.py --root . approve-codex --candidate-id kbc_xxx
python tools/knowledge_candidates.py --root . reject --candidate-id kbc_xxx --reason "not enough evidence"
python tools/knowledge_candidates.py --root . merge --candidate-id kbc_xxx
```

The report lists candidates by status and record type and exposes `ready_for_human_merge`, but it still does not merge records automatically. The `merge` command is intentionally a controlled block: it shows whether the candidate is ready and reminds the operator that active KB edits must be performed as an explicit human-reviewed source change.

## External Imports

Structured external imports are allowed only into staged candidates.

Implemented importer:

```bash
python tools/import_knowledge_sources.py --root . --pypi pydantic jsonschema
python tools/import_knowledge_sources.py --root . --official-doc "https://docs.python.org/3/library/csv.html|What does csv provide?|csv_table_transform"
```

Supported sources:

- PyPI JSON metadata: package summary, classifiers, project URLs, dependencies and keywords become weak archetype candidates.
- Official docs fetch: allowlisted documentation excerpts become weak `official_docs_fact` candidates.

PyPI candidates require the package/project name as a required match signal. This prevents generic words like `schema`, `token`, `client`, or `async` from becoming broad active heuristics.
